# Echo — Quality Assurance Audit

**Date:** 2026-06-18
**Scope:** Security and data integrity, performance and memory, architecture and best practices, test coverage, build and release configuration, and the Python seeding scripts.
**Deliverable:** This is a report only. No code has been changed. Everything below is a recommendation for you to review.

## How this audit was done

The codebase was reviewed by a set of specialist passes, each focused on a single area (security rules, authentication, Compose performance, coroutine and listener lifecycle, Firestore cost, architecture, tests, build configuration, and the seeding scripts). Every concern that a pass raised was then independently re-checked against the actual code by a second reviewer whose job was to be skeptical and throw out anything that wasn't genuinely a problem or that had been overstated. As a result, the findings below are the ones that survived scrutiny; a number of initial concerns were dismissed and are listed honestly at the end so they don't get raised again later.

The findings are grouped by how urgent they are. The launch blockers in the first section are things that will actively prevent you from publishing or that put the whole project at risk, and they should be dealt with before anything else. After that come the medium-severity issues (real bugs and real costs that are worth fixing but won't stop a release), then the lower-severity cleanup items, then a separate discussion of test coverage, and finally the list of concerns that were investigated and dismissed.

## 1. Launch blockers — these must be resolved before publishing

### The application ID uses the `com.example` prefix, which the Play Store will not accept

In [app/build.gradle.kts](app/build.gradle.kts#L28) the `namespace` and the `applicationId` are both set to `com.example.echo`. Google Play explicitly rejects any application ID that begins with `com.example`, because that prefix is reserved for sample and tutorial code. This matters more than a typical configuration tweak because the application ID is permanent: once an app is published under a given ID it can never be changed, so this has to be corrected before the very first upload, not afterward. You'll need to pick a real reverse-domain identifier that you control — something like `com.yourname.echo` — and then update the `namespace`, the package name inside `google-services.json`, the app registration in the Firebase console, and the signing certificate fingerprints to match. This was the single most important thing the audit found, and it was not on the project's existing roadmap.

### A live Firebase admin key is sitting unprotected inside OneDrive

The file at `C:/Users/Paul/OneDrive/Desktop/Echo/firebase-key.json` is a genuine Google service-account private key: it contains a real private-key PEM block and the service-account address `firebase-adminsdk-...@echo-2b5ba.iam.gserviceaccount.com`. The reason this is the most severe finding in the audit is that a service-account key of this kind bypasses your Firestore security rules and App Check entirely — it has full administrative access to the project's data. The folder it lives in is not a git repository and has no `.gitignore`, but more importantly it sits inside a OneDrive path, which means the key is being continuously synced up to Microsoft's cloud. Anyone who gains access to that OneDrive account, or any machine it syncs to, effectively owns your entire backend. You should revoke and regenerate this key in the Firebase console as soon as possible, and store the replacement somewhere local that is not synced to any cloud service and has restricted file permissions. The three Python scripts (`seed_pois.py`, `backfill_post_geohashes.py`, and `backfill_post_counts.py`) all load this key, so they'll need to point at the new location.

### Any signed-in user, including an anonymous guest, can read every user's email address

The `users/{uid}` documents store each account's real email address, and the read rule in [firestore.rules](firestore.rules#L149) is simply `allow read: if isSignedIn()`. The problem is that "signed in" includes anonymous guest sessions, which your app hands out freely through the "Continue as guest" button. That means anyone at all can open the app as a guest and then read any user's profile document — or list the whole collection — and collect every registered user's email address. The app itself never needs another user's email in order to function; the screens that display people only use the username, first name, last name, and bio. For a public social app heading to the Play Store this is a real privacy problem and the kind of thing a data-safety review will flag, because the emails were collected to support authentication, not to be handed out to other users. The cleanest fix is to stop keeping the email in the publicly-readable profile document at all: either move it into a private sub-document (for example `users/{uid}/private/contact`) that only the owner can read, or keep a separate display-only `publicProfiles/{uid}` document that contains no email. Either way the `email` field should also be removed from `UserProfileEntity` and `UserProfile` so it stops being downloaded by other clients. It's also worth considering whether guests need to read profiles at all, and tightening that rule to non-anonymous users if they don't.

### Users who signed in with Google cannot delete their account from inside the app

Google Play requires that every account, regardless of how it was created, can be deleted from within the app. Echo's data and view-model layers actually support this already — there is a Google re-authentication path and a `deleteAccountWithGoogle` method ready to use — but the user interface never reaches it. In [SettingsScreen.kt](app/src/main/java/com/example/echo/ui/settings/SettingsScreen.kt#L171), when the user confirms account deletion, the code branches on the sign-in provider, and the branch that covers Google accounts simply shows a message saying "Deleting a Google-linked account isn't available yet. Please contact support." Because a working in-app deletion path is a hard Play requirement, this needs to be wired up: connect the Google branch to the existing `deleteAccountWithGoogle` method (which performs the Credential Manager re-authentication) and then test the flow end to end. This is also entangled with the note in the project's memory that Google Sign-In itself isn't fully configured yet, so the two should be sorted out together.

## 2. Medium-severity issues — real problems worth fixing, but not release blockers

### The comment counter on any post or POI can be inflated or driven negative by anyone

In [firestore.rules](firestore.rules#L108) the rule that lets a client update `commentCount` only checks that the value moves by exactly plus or minus one in a single write, and it only requires that the caller is a non-anonymous user. It does not require that the caller owns the post, and it is not tied in any way to a comment actually being created or deleted. There is also no floor keeping the value at zero or above. The practical consequence is that any logged-in user can sit in a loop and repeatedly send "increment by one" writes against any post or any point of interest — including ones they don't own — to inflate the displayed comment count without limit, or send "decrement by one" writes to push it below zero. The damage is limited to a cosmetic counter rather than real content, which is why this is medium rather than high, but it's trivially abusable as vandalism. The robust fix, now that the project is on the Blaze plan, is to maintain `commentCount` from a Cloud Function that triggers when comments are created or deleted, and to forbid clients from writing the field directly. As a quick interim hardening you could at least require that the resulting value is zero or greater, though that alone won't stop the inflation.

### The feed list doesn't give its items stable identities, which causes visual glitches and wasted work

The main feed in [FeedScreen.kt](app/src/main/java/com/example/echo/ui/feed/FeedScreen.kt#L227) renders its posts with `items(uiState.posts) { ... }` and does not supply a `key`. Without a key, Compose identifies each row purely by its position in the list. That's fine as long as the list never changes shape, but the feed changes constantly — likes toggle, a refresh reorders or prepends posts, a new page is appended, and blocked authors' posts are removed. When the list shifts, Compose can no longer tell which row corresponds to which post, so it reattaches each row's internal remembered state to the wrong post. In practice that means the brief like-animation scale and the open/closed state of the overflow menu, both of which `PostCard` remembers per row, can jump onto the wrong item, and Compose also re-lays-out rows that didn't actually change. The rest of the app already does this correctly — the profile and post-detail lists both pass a stable key — so the feed, which is the largest and most dynamic list in the app, is the inconsistent one. The fix is small and safe: pass `key = { it.id }` to the `items` call, since a post's ID is a non-empty string and makes a valid stable key.

### Toggling a like reads the document and then writes it back, which is both racy and slightly wasteful

The `toggleLike` implementation in [PostRepositoryImpl.kt](app/src/main/java/com/example/echo/data/repository/PostRepositoryImpl.kt#L301) first fetches the post document, computes the new likes array in app code, and then writes it back — two separate round trips that are not wrapped in a transaction. There are two consequences. The first is correctness: if two different people like the same post at roughly the same moment, they both read the same "before" state, and the second write overwrites the first, so one of the likes is silently lost and the denormalized `likeCount` can end up permanently wrong. The code comment describing this as a single atomic write is misleading — it refers only to the two fields being set in one update call, not to the read and the write being atomic with respect to each other. The second consequence is cost: every like incurs an extra billed document read just to fetch the current array, and liking is a high-frequency action. The fix is to wrap the read and the write together in `firestore.runTransaction { }`. It's worth noting that the obvious shortcut of using `FieldValue.arrayUnion` and `FieldValue.increment` will not work here, because your own security rules validate a like by inspecting the concrete resulting array and asserting that `likeCount` equals the array's size — and those opaque server-side operators can't be checked that way. A transaction (which still performs the read, but safely) is the approach that's compatible with the rules you already have.

### Filtering the feed by tag reads the entire posts collection every time

The `getPostsByTag` method in [PostRepositoryImpl.kt](app/src/main/java/com/example/echo/data/repository/PostRepositoryImpl.kt#L125) attaches a live listener ordered by timestamp with no limit, and then filters by tag in Kotlin after the fact. This means that every time a user taps a tag, the app downloads and is billed for a read on every single document in the posts collection, and then throws away the ones that don't match. The pagination and 200-document cap that were added elsewhere to control read costs never reached this path. It's gated behind a deliberate tag tap and the tag view is short-lived, which keeps it at medium rather than high, but it scales badly as the collection grows. The better approach is to normalize tags to a consistent case when posts are written and then filter on the server with `whereArrayContains("tags", tag.lowercase())` together with a `limit`, so that only matching documents are downloaded and billed. Since the tag view is rare and short-lived, a one-time fetch would also be more appropriate than a live listener here. (For the record, only the feed's tag filter uses this method; the map does not, contrary to what an earlier note suggested.)

### The blocked-users stream opens its own duplicate Firestore listeners in every screen that uses it

The `observeHiddenUserIds` method in [UserRepositoryImpl.kt](app/src/main/java/com/example/echo/data/repository/UserRepositoryImpl.kt#L206) combines two raw Firestore listener flows but never shares the result — unlike the posts flow, which is deliberately shared across collectors. This method has four consumers: the map, feed, post-detail, and POI-detail view models. Because nothing shares the upstream, each of those view models that's alive opens its own independent pair of snapshot listeners on the same underlying data, one of which is a collection query. So when, for example, the feed and the map are both active, you're paying for two or more redundant sets of listeners watching identical data. The fix is the same pattern already used for posts: share the flow once (using `shareIn` on the application scope with a "while subscribed" policy) so that all four consumers collect from a single warm listener instead of each spinning up their own.

### The map re-runs a batch of geohash queries every time the camera stops moving

The map fetches nearby posts by issuing a batch of geohash range queries — typically around nine of them — for the current viewport. The trouble, visible in [MapViewModel.kt](app/src/main/java/com/example/echo/ui/maps/MapViewModel.kt), is that this happens every time the camera settles, and the only thing preventing a refetch is an exact-equality check on the bounds. Because the bounds change with even a small pan, almost any movement of the map produces a brand-new set of bounds and therefore a fresh, fully-billed batch of queries, with no debounce and no recognition that the new viewport largely overlaps the previous one. There are some mitigations already in place — the fetch is gated by zoom level, and the bounds are padded by about thirty percent so small pans don't always re-fire — but the underlying pattern still bills more reads than it needs to as the user explores the map. Adding a short debounce and some awareness of overlapping viewports would meaningfully reduce the read volume.

### Test-only navigation libraries are being shipped inside the production app

In [app/build.gradle.kts](app/build.gradle.kts#L110) the two navigation-testing libraries are declared with `implementation` rather than `androidTestImplementation`. Those artifacts contain test scaffolding such as `TestNavHostController` that has no purpose in a shipping app, so right now that test code is being bundled into the release APK that users download. Moving both declarations to `androidTestImplementation` keeps them available to your instrumented tests while removing them from the production build. (R8 shrinking will strip the unreferenced classes, so the real-world size impact is limited, but they still don't belong on the production classpath.)

### Repositories report failures inconsistently — some return a Result, others throw

Across the repository layer there's no single convention for how a failure is communicated. The comment repository throws exceptions on failure, the user and auth repositories consistently wrap outcomes in `Result`, and the post repository does both depending on the method — `createPost` returns a `Result` while `updatePost`, `deletePost`, and `toggleLike` throw. The interfaces involved are [CommentRepository.kt](app/src/main/java/com/example/echo/domain/repository/CommentRepository.kt) and [PostRepository.kt](app/src/main/java/com/example/echo/domain/repository/PostRepository.kt). This inconsistency forces every call site to remember which style a given method uses, which is easy to get wrong and tends to lead to swallowed errors. Settling on one convention — most likely `Result` everywhere, given how much of the code already uses it — would make error handling more predictable.

### Authentication errors are wrapped in a way that throws away the original exception

When the auth repository catches a Firebase exception, it builds a brand-new generic `Exception` carrying only a mapped message string, without passing the original exception along as the cause. This happens in several places in [AuthRepositoryImpl.kt](app/src/main/java/com/example/echo/data/repository/AuthRepositoryImpl.kt#L54). The downside is that any code further up the stack that wants to react to a *specific* kind of failure — for instance the `DeleteAccountUseCase`, which checks whether the error is a "recent login required" exception — can no longer see the underlying Firebase type, because it's been collapsed into a plain message. Preserving the original exception as the cause (or returning typed domain errors) would let callers handle different failures differently instead of only being able to show a string.

### Post coordinates are accepted without validation, so location can be spoofed (but this is a known, structural limit)

The post-creation rule in [firestore.rules](firestore.rules#L77) lists `latitude`, `longitude`, and `geohash` among the allowed fields but never checks their type or range, so a client can write whatever coordinates it likes — which means the five-kilometre proximity rule and a post's position on the map can be faked by talking to the database directly. I'm including this for completeness, but it's worth being clear that the rules file itself already documents this and explains why: Firestore rules have no trusted source for where the caller physically is, so this genuinely cannot be enforced in rules alone. It's the same problem your roadmap already plans to address with a Cloud Function (the server-side distance check), and it should be tracked there rather than treated as a rules bug.

## 3. Lower-severity issues — cleanup and hardening

### Security and rules

The comment-delete rules in [firestore.rules](firestore.rules#L118) permit any signed-in user, whereas the matching comment-create rule requires a non-anonymous user. In practice this can't be exploited today, because an anonymous user can never own a comment in the first place, but it's an inconsistent gate that would quietly become a hole if anonymous commenting were ever enabled, so it's worth aligning the delete rule to require a non-anonymous user like the create rule does.

The username reservation rule is less of a problem than it first appeared. Reservations are in fact stamped with the reserving user's own ID, so the original worry about one person squatting on handles tied to *other* people doesn't hold. What remains is only that a single user could reserve more handles for themselves than they should, because Firestore rules can't cheaply enforce the link between the reservation document and the profile document in a single transaction. It's a minor concern.

The report-submission rule in [firestore.rules](firestore.rules#L206) accepts an empty or arbitrary `reason` and `targetAuthorId`, because unlike the `targetId` field these aren't checked for a non-empty value. Since the reports collection is write-only and triaged by a human out of band, the worst case here is some noise in the moderation queue rather than any security or integrity issue.

The password-reset flow returns the raw Firebase error message to the user, while the rest of the auth flows map errors to friendly text. The relevant code is in [AuthRepositoryImpl.kt](app/src/main/java/com/example/echo/data/repository/AuthRepositoryImpl.kt#L103). Because Firebase distinguishes "no account exists for this email" from other errors, surfacing that raw message lets someone probe which email addresses have accounts. Returning a generic "if an account exists, we've sent a link" message regardless of outcome would close that small enumeration gap.

### Security and hygiene

The manifest enables `allowBackup` while the backup and data-extraction rule files are still the empty IDE stubs, as seen in [AndroidManifest.xml](app/src/main/AndroidManifest.xml#L11). With no rules in effect, all of the app's private data is eligible for cloud backup and device-transfer, which includes the Firebase authentication token store. It's worth being precise about the scope: the app's own preferences store holds only the theme, notification, and last-sync values, with no personal information, so the genuine concern is the auth session token rather than broad data leakage. Either turning off `allowBackup` or filling in the rule files to exclude the auth and datastore files would address it.

The secrets-default file is gitignored rather than committed, which means a fresh checkout or a CI machine won't have it, and the Maps API key placeholder will resolve to nothing — producing a blank map at runtime with no obvious build error. The relevant configuration is in [app/build.gradle.kts](app/build.gradle.kts#L31). Committing `local.defaults.properties` with just the placeholder value (while keeping the real key only in the gitignored `secrets.properties`) would make clean builds reproducible.

### Performance and memory

The alerts list in [AlertsScreen.kt](app/src/main/java/com/example/echo/ui/alerts/AlertsScreen.kt#L84) has the same missing-key issue as the feed, though the impact is lower because the alert cards don't hold any internal remembered state. Adding `key = { it.postId }` brings it in line with the rest of the app.

There's a dead full-collection listener, `getPostsWithLocation`, in [PostRepositoryImpl.kt](app/src/main/java/com/example/echo/data/repository/PostRepositoryImpl.kt#L148). It scans the entire posts collection but has no callers anywhere — the map now uses the geohash path instead — so it's a latent cost trap that should simply be deleted along with its use-case wrapper.

The `getProfilesByIds` method in [UserRepositoryImpl.kt](app/src/main/java/com/example/echo/data/repository/UserRepositoryImpl.kt#L134) fetches profiles one at a time in sequence rather than concurrently. This only runs when the blocked-users set actually changes, which is rare and usually involves very few users, so it's minor, but fetching them with a batched `whereIn` query or in parallel would be faster.

Comment creation and deletion perform two separate writes — the comment document and then the counter increment — without a batch or transaction, in [CommentRepositoryImpl.kt](app/src/main/java/com/example/echo/data/repository/CommentRepositoryImpl.kt#L85) and three sibling methods. If the process dies between the two writes, the comment exists but the count doesn't reflect it (or vice versa on delete), so the count can drift. Wrapping the pair in a `WriteBatch` makes them commit together without changing the number of billed writes.

Throughout the UI, screens collect their state with `collectAsState()` rather than `collectAsStateWithLifecycle()`. The consequence is subtle but real: because plain collection isn't lifecycle-aware, a backgrounded screen keeps collecting, so the "while subscribed" timeout never sees its subscriber count drop to zero, and the shared Firestore listeners stay warm and keep billing while the app is in the background. The cost is modest — these are listeners on already-cached data with a short grace period — but switching to the lifecycle-aware collector across the screens would let the listeners pause properly when the app isn't visible.

### Architecture and consistency

A few clean-architecture boundaries are crossed. The `DeleteAccountUseCase` in the domain layer imports and matches on concrete Firebase exception types, which means the domain layer depends on the Firebase SDK; and several view models (feed, post-detail, POI-detail) inject `FirebaseAuth` directly to read the current user ID instead of going through a repository. Neither causes a bug today, but both are the kind of leak that makes the layers harder to test and to reason about. There's also some genuinely dead code worth removing: `getPostsWithLocation`, `getPostsByUsername`, the no-op `refreshPosts` on the repository, `PoiMapper.toEntity`, and `GetPostByIdUseCase` are all unreferenced. (Note that `getPosts` is *not* dead — the map and a test still use it.)

On consistency, several Firestore field names are hardcoded as string literals even though there's a `Constants` registry meant to hold them, and some fields have no constant at all. Separately, `UserProfile` is mapped to and from its entity with a private inline extension function, while every other entity uses a dedicated mapper class in the `data/mapper` package, so the user profile is the odd one out. Finally, [architecture.md](architecture.md) has drifted from the code: it describes a `data/model` package containing `Dto` classes, but the code actually uses a `data/entity` package with `Entity` classes. Updating the document would keep it trustworthy as the stated source of truth.

### Build and release

When `keystore.properties` is absent, the release build silently falls back to the debug signing key, as configured in [app/build.gradle.kts](app/build.gradle.kts#L73). This won't reach users — the Play Console rejects a debug-signed bundle at upload time — but a developer could produce a release bundle that's quietly debug-signed without realizing it. Making the release build fail outright when the keystore is missing would remove that surprise. Relatedly, the navigation dependencies in [libs.versions.toml](gradle/libs.versions.toml#L17) mix a release-candidate version (`2.9.0-rc01`) with stable ones (`2.8.9`); because Gradle resolves the group to the highest version, the launch build ends up shipping the release-candidate navigation library. Aligning all of them to a single stable version avoids shipping pre-release code.

### The seeding scripts

The `seed_pois.py` script is destructive if it's run more than once. It deletes all the existing POI documents and then re-adds new ones with fresh auto-generated IDs. Because Firestore does not delete sub-collections when you delete a parent document, any existing `comments` under the old POIs are left orphaned, disconnected from the new POI documents. (The earlier description of this as "wiping the comment count to zero" was slightly off — the POI documents don't actually carry a comment-count field — but the orphaning is real.) Making the script upsert by a stable ID rather than delete-and-recreate would fix this. Two smaller notes on the same script: it writes `createdAt` using a naive `datetime.utcnow()` rather than the server timestamp the app uses elsewhere, so it depends on the seeding machine's clock; and it calls the Wikimedia image API once per POI with no retry, backoff, or throttling. The request volume is low (around forty sequential calls), so the lack of throttling isn't an operational risk, but there's no resilience if a call fails.

## 4. Test coverage

This is less a list of individual bugs than a single observation: the most important logic in the app is essentially untested. There are six real test files, and the highest-risk areas have no coverage at all.

The map's clustering algorithm is both the most complex piece of logic in the app and completely untested, and it's also hard to test as written because it operates directly on Google Maps' `LatLng` type and computes distances with Android's `Location.distanceBetween`, both of which don't work in a plain JVM unit test without Robolectric. Extracting the clustering into a pure function that doesn't depend on Android types would make it both testable and easier to reason about. The `FeedViewModel` has a test file, but it only covers the initial location fetch — the pagination, the optimistic like update, and the block and report flows are all untested. There are no repository tests whatsoever, so the ownership and guest guards on creating posts, toggling likes, and adding comments are unverified by anything but the security rules. The validation helpers are untested even though the password and username checks are simple, pure regex that would be trivial to cover (only the email check, which leans on `android.util.Patterns`, would need Robolectric). The pure formatting helpers for timestamps and distances are untested despite being easy wins. And there's no screenshot-testing framework set up; the Compose UI-test dependencies are present but no UI tests have actually been written, while the two example test files are tautological template stubs that should just be deleted.

A sensible order to build coverage would be to start with the easy, high-value pure functions (the formatting and validation helpers, then the mappers), move on to the repository guard logic, then the `FeedViewModel` behavior, and finally tackle the clustering once it's been extracted into a testable form.

## 5. Memory leaks, runtime leak detection, and R8 optimization

This section covers a dedicated memory pass that goes beyond the listener-lifecycle notes elsewhere in the report: a static search for leaks, the addition of runtime leak detection, and a review of the R8 shrinking configuration.

### The static leak analysis came back clean

The most important result is a negative one: there is no structural memory leak to fix. Every Firestore snapshot listener in the app is removed when its flow is cancelled — all eight `callbackFlow` sites across the post, comment, POI, and user repositories end with `awaitClose { listener.remove() }`, and the reactive authentication listener is handled the same way, with `AuthRepositoryImpl.authState()` calling `awaitClose { auth.removeAuthStateListener(listener) }` at [AuthRepositoryImpl.kt:36](app/src/main/java/com/example/echo/data/repository/AuthRepositoryImpl.kt#L36). Unremoved listeners are the most common source of both leaks and runaway billing in a Firestore app, and they are handled correctly throughout. Beyond that, the location provider is a singleton that holds only the application context rather than an Activity ([LocationProviderImpl.kt:31](app/src/main/java/com/example/echo/data/location/LocationProviderImpl.kt#L31)), there is no use of `GlobalScope` or any static mutable Context, and the view models keep to `viewModelScope` and inject the application context or `FirebaseAuth` rather than holding a View or Activity. The application-scoped coroutine scope is a deliberate singleton, not a leak.

The only two things worth noting, and neither is a leak, are the process-lifetime bitmap caches `iconCache` in [MapUtils.kt:17](app/src/main/java/com/example/echo/ui/maps/MapUtils.kt#L17) and `pinCache` in [CreatePinIcon.kt:65](app/src/main/java/com/example/echo/ui/maps/CreatePinIcon.kt#L65). They keep rendered marker icons for the life of the process, but they store `BitmapDescriptor` objects (which do not retain a Context) and their key spaces are bounded, so they are fine as written. The only thing to watch is that the cache keys never come to include a continuous value such as a raw color integer, which would let them grow larger than intended.

### LeakCanary was added to catch the dynamic leaks static analysis can't see

A static pass can only find structural problems; it cannot catch the leaks that appear when the app is actually used — a Composition that is retained, a navigation back-stack entry that won't release, a captured callback that outlives its screen. To cover that, LeakCanary was added temporarily as a debug-only dependency (`debugImplementation("com.squareup.leakcanary:leakcanary-android:2.14")`). It installs itself automatically through its own ContentProvider, so no application code was needed, and because it was a `debugImplementation` it was never compiled into the release build.

It was then run on a real device. The debug build was installed on a Samsung SM-F731U (a Galaxy Z Flip 5 foldable), and the leak-prone flows were exercised under the harshest realistic conditions — the "Don't keep activities" developer option was enabled so that every screen exit destroys the Activity, and the app was driven through repeated Feed-to-Map-to-Profile tab switches and repeated open-and-back cycles through post detail, after which it was sent to the background to trigger LeakCanary's retained-object check. LeakCanary reported only that it was "running and ready to detect memory leaks" and never escalated to a heap dump, which is its behaviour when nothing is being retained. Confirming that on disk, the app's storage contained no `.hprof` heap dump and LeakCanary's `leaks.db` held no leak record referencing any `com.example.echo` class. In other words, the runtime check agreed with the static analysis: no leak was found.

Having served its purpose, the LeakCanary dependency was then removed from [app/build.gradle.kts](app/build.gradle.kts) and a clean debug build reinstalled, so it is no longer present in the project or on the device. It can be re-added with that single `debugImplementation` line whenever a future change warrants another leak-detection pass before launch.

### R8 keep-rule configuration

The release build is set up correctly — minification, resource shrinking, and the optimized default ProGuard file are all enabled, and R8 full mode is on by default. Two safe improvements were identified and have now been applied. First, there was a redundant keep rule: in [app/proguard-rules.pro](app/proguard-rules.pro) the rule `-keepclassmembers class com.example.echo.data.entity.** { *; }` was fully subsumed by the `-keep class com.example.echo.data.entity.** { *; }` immediately above it, because keeping a class with all members already keeps every member. That redundant line has been deleted, leaving the single `-keep class` rule in place. Second, `android.r8.optimizedResourceShrinking=true` has been added to [gradle.properties](gradle.properties); this enables the newer, more effective resource shrinker that AGP versions between 8.6 and 9.0 (the project is on 8.11.1) must opt into explicitly, and which becomes the default in AGP 9.0.

One further refinement was identified but deliberately not applied. The remaining entity keep rule is a broad, package-wide, all-members rule that blocks R8 from shrinking, optimizing, or obfuscating any of the Firestore entities. It exists because Firestore deserializes those classes reflectively, so it cannot just be removed, but it could be narrowed to `-keepclassmembers class com.example.echo.data.entity.** { <init>(); <fields>; }`. That narrowing has a real caveat: Firestore maps document keys to fields by name, so field-name obfuscation must stay disabled (which the narrower rule preserves) unless every field is annotated with `@PropertyName`. Because getting this wrong silently breaks deserialization — objects come back empty — it should only be applied alongside a release-build round-trip read test, which is why it was left for a follow-up rather than changed blind. Separately, upgrading to AGP 9.0 would bring faster builds and further R8 optimizations, though that is larger, separate work.

## 6. Concerns that were investigated and dismissed

A few things were raised during the audit and then disproven on closer inspection. I'm recording them here so they don't get re-raised.

The concern that account deletion removes the Firestore profile before the auth account, leaving an orphaned auth user, turned out to be backwards. The actual deletion flow re-authenticates first, then deletes the username and profile documents, and deletes the Firebase auth user *last* — which is the correct order. Only the opposite and much rarer edge case exists (the Firestore documents are deleted and then the auth deletion fails), and that case self-heals through the best-effort error handling on a subsequent attempt.

The concern that the Google Maps API key is committed unrestricted and could be used to run up billing was also dismissed. The key is restricted to the Android app by package name and signing certificate; what was removed earlier was only the API-*type* restriction, which had been breaking token refresh. The app restriction is exactly what prevents someone from lifting the key out of the APK and replaying it. The only residual item is administrative: the release and Play app-signing certificate fingerprints need to be registered before launch, and optionally you could split out a dedicated Maps key with a budget alert.

The concern that the application-scoped coroutine scope is never cancelled and therefore leaks was dismissed as well — that scope is intentional and is the correct host for the app-wide shared flows; the "while subscribed" policy tears the underlying listeners down when nothing is collecting, so nothing is held indefinitely. And the storage rules and empty indexes file are both correct as they stand: the deny-all storage rule is appropriate because there's no storage feature yet, and the empty indexes file matches all of the app's current single-field queries.

## 7. Suggested order of work

The launch blockers in section 1 should come first, because they either prevent publishing outright or put the whole backend at risk: rename the application ID, rotate and relocate the service-account key, separate the email field out of the publicly-readable profile document, and wire up Google-account deletion. After that, the cheapest high-confidence wins are worth doing as a batch — adding the missing list keys to the feed and alerts, adding the zero-floor to the comment-count rule, aligning the anonymous comment-delete gate, moving the test-only navigation libraries off the production classpath, and deleting the confirmed dead code. The next tier is robustness: making the like toggle a transaction, batching the comment writes, sharing the blocked-users flow, bounding and debouncing the tag and map queries, fixing the password-reset message, and moving the screens to lifecycle-aware state collection. After that come the remaining hardening and consistency items — the backup rules, the secrets default file, the error-handling conventions, the layering leaks, the hardcoded field names, the architecture document, the script idempotency, and the navigation version alignment. Finally, standing up real test coverage in the order described in section 4 will raise the project to the "testable" bar that the architecture document already claims for it.
